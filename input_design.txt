# BounceRL Input System Design

## Overview

The BounceRL input system takes in high-level PyAutoGUI style input actions either directly, or through a Gym action space, and dispatches raw input events to a Desktop backend.

To ensure models trained to output gym actions can be used across environments or BounceRL versions that support different sets of inputs, we do two things:

1. We use a fixed Gym action space across all apps/environments and mask out the actions corresponding to disallowed inputs on a per-environment basis.

2. We'll keep the gym action space to desktop input mapping stable, only append new inputs to the end of the action space, and make it easy to no-op pad a smaller action space action to be in a newer, larger, action space.

We don't support actions like `type("hello world")`, but it might not be too hard to add if/when necessary.

We also don't support mouse zoom yet, but we'll add it as a fast follow.

## keys.py

Defines the set of keys our input system will use, along with "key classes" — sets of related keys one may want to use, for example, to specify which keys an application environment allows agents to use.

### Keys

All of the keys use X11 keysym values. Bouncer keys are uncased whereas X11 keysyms are cased. We use the uppercase keysyms for the letters and can convert to lowercase keysyms by adding `LOWERCASE_OFFSET`.

```python
LOWERCASE_OFFSET = 0x0020

# Letters
KEY_A = 0x0041
KEY_B = 0x0042
# ...

# Digits
KEY_0 = 0x0030
# ...

# Punctuation
KEY_EXCLAM = 0x0021      # !
KEY_AT = 0x0040          # @
KEY_NUMBERSIGN = 0x0023  # #
# ...

# FnKeys
# ...

# Modifiers
# Note: We only implement the left shift, alt, control keys.
KEY_SHIFT_L = 0xffe1
KEY_ALT_L = 0xffe9
KEY_CONTROL_L = 0xffe3

# Other
# Tab, Escape, Enter, Backspace
# ...

# Pointer Buttons
KEY_POINTER_BUTTON1 = 0xfee9
KEY_POINTER_BUTTON2 = # ...
KEY_POINTER_BUTTON3 = # ...
LEFT_MOUSE_BUTTON = KEY_POINTER_BUTTON1
RIGHT_MOUSE_BUTTON = KEY_POINTER_BUTTON2
MIDDLE_MOUSE_BUTTON = KEY_POINTER_BUTTON3
SCROLL_UP_BUTTON = KEY_POINTER_BUTTON4
SCROLL_DOWN_BUTTON = KEY_POINTER_BUTTON5
```

### Key Classes

```python
Letters = (KEY_A, ..., KEY_Z)
Digits = (KEY_0, ..., KEY_9)
Punctuation = (KEY_EXCLAM, KEY_AT, KEY_NUMBERSIGN, ...)
    # Keys: ! @ # $ % ^ & * ( ) - _ = + [ ] { } \ | ~ ` ; : ' " , < > . ? /
FnKeys = (KEY_F1, ..., KEY_F12)
Modifiers = (KEY_SHIFT_L, KEY_ALT_L, KEY_CONTROL_L)  # No right-side modifiers
Other = (KEY_TAB, KEY_ESCAPE, KEY_ENTER, KEY_BACKSPACE)
MouseButtons = (KEY_POINTER_BUTTON1, KEY_POINTER_BUTTON2, KEY_POINTER_BUTTON3)
ScrollButtons = (KEY_POINTER_BUTTON4, KEY_POINTER_BUTTON5)
```

The rest of the keys are unmapped.

> **Note:** As per [vncdotool RFB protocol docs](https://vncdotool.readthedocs.io/en/78/rfbproto.html) KeyEvent section:
> - In the RFB protocol, shift key state is a hint; letter keysyms are authoritative for letter casing.
> - On the other hand, ctrl and alt key state in the protocol is authoritative for whether or not a key is modified by ctrl or alt.

## allowed_inputs.py

Allowed input mappings can be constructed as allow lists or deny lists of keys and key classes.

**Example:** The Factorio allowed mapping is:
```python
DisallowKeys([FnKeys, Escape])
# or equivalently:
AllowKeys([Letters, Digits, Punctuation, Modifiers, Tab, Enter, Backspace, MouseButtons])
```

### Classes

```python
class AllowKeys(List[Key | KeyClass]):
    def keys(self) -> List[Key]:
        """Returns the expanded list of allowed keys."""

class DisallowKeys(List[Key | KeyClass]):
    def to_allow_list(self) -> AllowKeys:
        """Takes all keys and removes disallowed ones."""
```

## input_types.py

### KeyAction

Represents a slightly higher level action for a key, like PyAutoGUI actions. The input system can convert actions to raw key up and key down events.

```python
@dataclass(frozen=True)
class KeyAction:
    action: KeyActionType  # KEY_PRESS, KEY_UP, KEY_DOWN
    key: Key
```

### KeyEvent

A raw key event. Our input system generates these and sends them to our desktop.

```python
@dataclass(frozen=True)
class KeyEvent:
    action: KeyDirection  # KEY_UP, KEY_DOWN
    key: Key
```

### MouseAction

Relative or absolute mouse position moves as well as drag events. Mouse coordinates are in screen pixels.

```python
@dataclass(frozen=True)
class MouseAction:
    is_relative: bool
    position: tuple[int, int]
    is_drag: bool
    drag_button: Key | None
```

**Factory functions:**
```python
def move_mouse(delta_position: tuple[int, int]) -> MouseAction: ...
def move_to(position: tuple[int, int]) -> MouseAction: ...
def drag_mouse(button: Key, delta_position: tuple[int, int]) -> MouseAction: ...
def drag_mouse_to(button: Key, position: tuple[int, int]) -> MouseAction: ...
```

### MouseEvent

A raw mouse move event. Our input system generates these and sends them to our Desktop. All mouse events are absolute position moves.

```python
@dataclass(frozen=True)
class MouseEvent:
    position: tuple[int, int]  # Absolute screen pixel position
```

### Union Types

```python
InputAction = KeyAction | MouseAction
InputEvent = KeyEvent | MouseEvent
```

## gym_input.py

Helpers to expose our input system through the Gym API. BounceRL exposes all KeyActions and MouseActions in a fixed layout action space and masks out disallowed actions when applying gym inputs to the desktop.

We'll keep our Action Space compatible across versions by adding new keys to the back of the action space.

### Action Space

The action space is a `Tuple` with three elements:

#### [0] MultiDiscrete - Key actions

One `Discrete(4)` per key in `ACTION_KEYS` (includes mouse buttons):

| Value | Action   |
|-------|----------|
| 0     | no-op    |
| 1     | press    |
| 2     | key_down |
| 3     | key_up   |

- Shape: `(len(ACTION_KEYS),)`
- *Future keys will be appended to ACTION_KEYS*

#### [1] MultiDiscrete - Mouse discrete actions

| Index | Component        | Type         | Values                          |
|-------|------------------|--------------|--------------------------------|
| 0     | Translation mode | Discrete(2)  | absolute=0, relative=1         |
| 1     | Action           | Discrete(3)  | none=0, move=1, drag=2         |
| 2     | Drag button      | Discrete(3)  | left=0, right=1, middle=2      |

- Shape: `(3,)`
- *Future mouse discrete components will be appended here*

#### [2] Box - Mouse position

- Shape: `(2,)` representing `[x, y]`
- Bounds: `[-400, -400]` to `[screen_width, screen_height]`
- Absolute positions are clamped to `[0, screen_width]` and `[0, screen_height]` by input event processing
- Relative positions are clamped to `[-400, 400]` for both x and y by input event processing
- *Future box/continuous components would be added as additional tuple elements*

#### Example

```
action[0] has shape (len(ACTION_KEYS),)  - all key actions
action[1] has shape (3,)                 - [translation_mode, mouse_action, drag_button]
action[2] has shape (2,)                 - [x, y] position
```

### Constants

```python
# Tuple indices

# Key action values

# Mouse discrete indices (within action[1])

# Mouse translation mode values

```

### Functions

#### `action_space(screen_width: int, screen_height: int) -> spaces.Tuple`

Return the Gym action space for inputs.

#### `mask_action(action: GymAction, allowed_inputs: AllowKeys) -> GymAction`

Masks the given Gym action to only allowed inputs. Environments apply this automatically, but users can also use this themselves to normalize their actions to just the inputs that apply to an environment.

Disallowed key actions are set to `KEY_NOOP` (0).

#### `process_gym_action(action: GymAction) -> List[InputAction]`

Converts a Gym action to its corresponding input actions.

Mouse position values are clamped during processing:
- Absolute positions: clamped to `[0, screen_width]` and `[0, screen_height]`
- Relative positions: clamped to `[-400, 400]` for both x and y

## input_processor.py

Manages keyboard and mouse state and generates events from actions.

### Keyboard State Management

- Generating correctly cased letter presses according to shift key state
- Releasing a held key before pressing it
- Ignoring key down and key up actions when the key's already in that position

### Mouse State Management

- Converting relative to absolute mouse coordinates based on the last mouse position
- Clamping the mouse position to within the screen rect

### Notes

- Any non-letter keys in our input system are unaffected by shift key state, so for example, an action sequence of "shift" then "1" will emit a "1", not an "!". This reflects how our Desktop backend (which uses the VNC/RFB protocol) works: punctuation characters have their own distinct keysyms (e.g., KEY_EXCLAM for "!"), so agents must output the punctuation keysym directly rather than shift+digit combinations.
- All modifiers are passed through directly as events, with ctrl and alt needing no special handling beyond direct event pass through, unlike the special handling we do for the shift key.
- The input_processor's mouse position is initialized at (0, 0), so an initial relative mouse move will be an absolute one relative to this position. This is fine, since macros should use absolute coordinates, and this single position-discontinuity will be insignificant to any agent that can effectively control a mouse.

### Class

```python
class InputProcessor:
    def __init__(self, screen_x: int, screen_y: int): ...
    
    def process_input_actions(self, actions: List[InputAction]) -> tuple[List[InputEvent], List[InputEvent]]:
        """
        Returns two lists of events:
        
        1. Events that callers should issue immediately
        2. Events that callers should issue after some delay if they want 
           polling-based apps to see input action intermediate button states
        
        For example: a keyPress action generates a keyDown event in the first 
        list and a keyUp event in the second. By issuing these events with a 
        small delay in between (say 50ms), polling-based apps have the chance 
        to see the key-down state before the key is released.
        
        Compound actions map as:
          KeyPress -> ([KeyDown], [KeyUp])
          MouseDrag -> ([ButtonDown, MoveMouse], [ButtonUp])
        """
    
    def release_buttons(self) -> List[KeyEvent]:
        """Releases any held buttons. Useful when resetting an environment."""
```

## benchmark.py

Test how much time it takes to process fixed Gym input actions and dispatch their generated events to a fake Desktop.

### FakeDesktop

Implements the same event interface as Bounce Desktop, with functions that do nothing.

### Benchmark Actions

`benchmark_action_a` and `benchmark_action_b` are fixed gym actions in our gym action space such that:

- For keys in the action space with keysym ≡ 0, 1 (mod 4): no-op
- For keys in the action space with keysym ≡ 2 (mod 4): press the key
- For keys in the action space with keysym ≡ 3 (mod 4): if input a, press down; if input b, release key
- For mouse: input a and input b both move the mouse relative (10, 10)

### benchmark_input_system

```python
def benchmark_input_system(loop_iters: int = 1_000_000) -> float:
    """
    Runs the end-to-end input system benchmark and returns
    the average time in seconds to process and dispatch a new input.
    """
    input_processor = InputProcessor()
    desktop = FakeDesktop()

    # Time this loop:
    for i in range(loop_iters):
        if i % 2 == 0:
            action = benchmark_action_a
        else:
            action = benchmark_action_b
        input_actions = process_gym_action(action)
        immediate_events, delayed_events = input_processor.process_input_actions(input_actions)
        apply_bounce_desktop_events(immediate_events, desktop)
        apply_bounce_desktop_events(delayed_events, desktop)

    return total_loop_time / loop_iters

if __name__ == "__main__":
    print(f"Action input handling takes {benchmark_input_system()} seconds")
```

### input_processor.py

#### Test Shift Casing

- **test_shift_before_letter_gives_uppercase**: Holding shift before a letter press gives an upper case letter press
  - Actions: `shift down`, `a down`
  - Should generate: immediate=`[shift key down, uppercase A down]`, delayed=`[]`

- **test_shift_after_letter_gives_lowercase**: Holding shift after a letter press gives a lower case letter press
  - Actions: `a down`, `shift down`
  - Should generate: immediate=`[lowercase a down, shift down]`, delayed=`[]`

- **test_release_uses_original_case**: Releasing a held key releases the key's original case
  - Actions: `shift down`, `a down`, `shift up`, `a up`
  - Should generate: immediate=`[shift down, uppercase A down, shift up, uppercase A up]`, delayed=`[]`

#### Test Key Up/Down State Handling

- **test_pressing_held_key_releases_first**: Pressing a held key generates a key-up before the press
  - Actions: `a down`, `press a`
  - Should generate: immediate=`[a down, a up, a down]`, delayed=`[a up]`

- **test_releasing_up_key_generates_nothing**: Releasing an already up key generates nothing
  - Actions: `a up`
  - Should generate: immediate=`[]`, delayed=`[]`

- **test_pressing_down_pressed_key_generates_nothing**: Pressing down a pressed key generates nothing
  - Actions: `a down`, `a down`
  - Should generate: immediate=`[a down]`, delayed=`[]` (just the first a down)

#### Test Two-List Return Mechanism

- **test_key_press_returns_immediate_and_delayed_events**: Verify that press actions correctly split into immediate down and delayed up
  - Actions: `press a`
  - Should generate: immediate=`[a down]`, delayed=`[a up]`

- **test_mouse_drag_returns_immediate_and_delayed_events**: Verify that drag actions correctly split events
  - Actions: `drag_mouse(LEFT_MOUSE_BUTTON, (10, 10))`
  - Should generate: immediate=`[left_button down, move_event(10, 10)]`, delayed=`[left_button up]`

#### Test release_buttons()

- **test_release_buttons_releases_held_keys**: Release buttons releases down buttons
  - Actions: `press a`, `b down`, `c down`, `d down`, `d up`
  - Then `release_buttons()` should return: `[b up, c up]` (only currently-held keys)

#### Test Mouse State Handling

- **test_mouse_moves_clamped_to_screen**: Absolute and relative mouse moves are clamped to staying on the screen
  - Create `InputProcessor(600, 600)`
  - Actions: `move_to(800, 800)`, `move_to(-200, -200)`, `move(1000, 0)`
  - Should generate: immediate=`[move_event(600, 600), move_event(0, 0), move_event(400, 0)]`, delayed=`[]`
  - Note: relative move clamped to [-400, 400] range, so move(1000, 0) becomes move(400, 0)

- **test_relative_moves_correct_position**: Relative moves go to correct location
  - Create `InputProcessor(600, 600)`
  - Actions: `move_to(100, 100)`, `move(10, 5)`, `move(20, 10)`
  - Should generate: immediate=`[move_event(100, 100), move_event(110, 105), move_event(130, 115)]`, delayed=`[]`
